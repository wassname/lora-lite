  naming)

  B = V[:, :r] @ diag(√Sr)                    # absorbs 1/√scaling

● 6. HRA review

  Paper (Yuan et al. 2024):
  W' = W·R,     R = ∏_{i=0..r-1} H_i,    H_i = I - 2·u_i·u_i^T / ||u_i||^2
  init (paper): independent random columns
  peft variant: paired columns (u_{2k}=u_{2k+1}) → R=I exactly at t=0, even r
  GS variant (apply_GS=True): explicit Gram-Schmidt for strict orthogonality
  Loss-side: orthogonality regularizer (Eq. 6); not part of layer.

    R = I_{d_in}
    for i in 0..r-1:                               # forward order
        u = opt_u[:, i]
        R = R - 2·R·u·u^T                          # R ← R · H_i
  forward(x):
    R = product over active adapters
    return F.linear(x, W @ R, bias)                # = x · R^T · W^T
  (effectively R applied to x)
  init: half_u = kaiming((d_in, r/2)); u = repeat_interleave(half_u, 2, dim=1)
   # cols 0=1, 2=3, ...

  ours src/lora_lite/variants/hra.py:53-75:
  lora_U: (r, d_in)                                # ROWS are vectors
  (transpose convention)
  forward_input(x):
    Rx = x
    for i = r-1 downto 0:                          # reverse order, applied
  directly
        u = U[i]
        sq = (u·u).sum().clamp_min(1e-12)
        coeff = (Rx · u) · (2/sq)
        Rx = Rx - coeff · u                        # Rx ← Rx · H_i
    return Rx                                       # then base layer applies W
  to this
  init: half = kaiming((r/2, d_in)); U = repeat_interleave(half, 2, dim=0)    #
  rows 0=1, 2=3, ...

  Diff vs peft:
  - ✓ Math equivalent. peft computes R = H_0·H_1·...·H_{r-1} (col-vector
  convention), then applies via x·R^T = x·H_{r-1}·...·H_0 (since H is
  symmetric). Ours iterates i = r-1 → 0 doing Rx ← Rx·H_i, computing
  x·H_{r-1}·...·H_0. Same.
  - ✓ symmetric init: row pairs vs col pairs — consistent with our row-vector
  convention.
  - ✓ identity at t=0: paired u_a=u_b → H_a·H_a=I (involution). R=I.
  - ⚠ No apply_GS=True path. peft offers Gram-Schmidt orthogonalization.

  ||opt_u||_col, so gradients flow through normalization. Ours keeps ||u||² in
  the denominator each step, gradients flow through that. Both define the same
  orthogonal H_i; gradients should give equivalent training to within numerical
  noise, but not bit-identical.
  - ✓ even-r requirement raised loudly (peft warns + falls back; we raise —
  preferable for fail-fast).

  Verdict: correct + faster than peft. Ordering bug from 67a6daf is properly
  fixed (verified: i = r-1 → 0 iteration matches peft's R = H_0..H_{r-1} then
  x·R^T).

● 8. AntiPaSTO review

  Reference (antipasto3, Flax):
  kernel = U·diag(S)·Vh + W_res             # kernel: (in, out)
  init U,S,Vh,W_res = svd(kernel) (frozen, fp32 / bf16 W_res)
  trainable:
    delta_s     ∼ truncated_normal·4e-4 + 4e-4   # small POSITIVE bias
  (sym-break)
    rot_T       ∼ truncated_normal·1e-4          # small random
  forward(x, alpha):
    A = scatter(rot_T into upper-triu of (n_blocks, bs, bs))
    A = 0.5·(A - A^T);  A = a_lim·tanh(A/a_lim)
    X = α·A/2                                     # << α scales rotation
    R = (I-X)^-1 (I+X)                            # block Cayley
    if rotate_U:  U_rot = U @ R                   # einsum, never builds full
  (r,r)
    if rotate_V:  Vh_rot = R @ Vh
    S_scaled = S + α·delta_s
    return (x @ U_rot)·S_scaled @ Vh_rot + x @ W_res
  Identity property: bidirectional symmetry — R(α=+1) and R(α=-1) are exact
  inverses.
  At α=0: U,S,Vh untouched → kernel reproduced.

  ours src/lora_lite/variants/antipasto.py:
  weight (pytorch): (d_out, d_in);  SVD: U:(d_out,r), S:(r,), Vh:(r,d_in)
  init: U,S,Vh frozen buffers; W_res mutates layer.weight (PiSSA-style)
  trainable:
    delta_s = 0
    rot_T   = 0                                   # << EXACT identity at t=0
  forward(x, y):
    A = scatter(rot_T)                            # (n_blocks, bs, bs)
    A = 0.5·(A - A^T);  A = a_lim·tanh(A/a_lim)
    X = A/2                                        # << no α
    R_blocks = Cayley(X)
    R = block_diag(R_blocks)                       # << builds full (r,r) via
  for-loop
    if rotate_basis=="V":  Vh_eff = R @ Vh
    elif rotate_basis=="U": U_eff = U @ R^T
    S_eff = S + delta_s
    h = (x @ Vh_eff^T) · S_eff
    delta = h @ U_eff^T
    return y + delta                               # y = x @ W_res^T already
  Identity at t=0: rot_T=0 → R=I, delta_s=0 → delta = x @ Vh^T @ diag(S) @ U^T =
   x @ W_svd^T → exact bit-identity.

  Diff vs antipasto3:

  - ⚠ No alpha steering coefficient. antipasto3's headline feature —
  bidirectional symmetry (R(+α)·R(-α)=I) — is dropped. If you intend antipasto
  as a fine-tuning-only adapter, this is a deliberate scope cut; if you ever
  want bidirectional steering at inference, you'd need to thread an α through
  the forward.
  - ⚠ Rotation chirality flipped. antipasto3 rotates the input basis by R (U @ 
  R); ours rotates by R^T (Vh_eff = R @ Vh ⇒ (x @ Vh^T) @ R^T). Same
  orthogonal-group element up to inversion, so gradient-equivalent under sign
  flip of rot_T. Not a correctness bug, but checkpoints aren't transferable
  between the two impls without a sign flip.
  - ✓ Stricter t=0 identity. Ours zero-inits both delta_s and rot_T → exact
  bit-identity. antipasto3 nudges delta_s ≈ +4e-4 and rot_T ∼ 1e-4 for symmetry
  breaking.
  Foot-gun: at exact zero rotation, ∂loss/∂rot_T may be sign-symmetric (rotating
   in either direction gives same loss to first order), so optimizer can stall
  on rot_T while delta_s carries the load. antipasto3's small bias avoids this.
  Worth a docstring note.
  - ✗ Perf: full (r,r) block-diagonal materialized via Python for-loop in
  _block_diag. antipasto3 uses einsum('dnb,nbc->dnc', U_reshaped, R_blocks) to
  apply each block directly without ever building the (r,r) matrix. For r=64,
  bs=4 this is 16 reflective blocks; the for-loop is O(r²) memory and Python
  overhead. Easy fix:
  # rotate V (input basis): replace block_diag + matmul with einsum
  Vh_blocks = Vh.reshape(n_blocks, bs, d_in)
  Vh_eff = einsum(R_blocks, Vh_blocks, "n a b, n b i -> n a i").reshape(r, d_in)
  - ⚠ rotate_basis enum ("V" XOR "U") vs. antipasto3's two booleans (can do
  both). Minor expressivity loss.
  - ✓ Cayley, skew construction, tanh angle clamping all match antipasto3
  exactly (incl. the 0.5 factor on (A - A^T)).
  - ✓ W_res via PiSSA-style in-place mutation of layer.weight is more
  memory-efficient than antipasto3's separate buffer; semantically identical.
  - ⚠ arxiv URL: https://arxiv.org/abs/2601.07473 — earlier I confirmed it
  returns HTTP 200, so the paper exists. Fine.

  Verdict: correct as a fine-tuning adapter with stricter t=0 identity. Two
  divergences from antipasto3 worth deciding on:

  1. Chirality: trivial sign-flip in skew if you want checkpoint compat with
  antipasto3 — otherwise leave.
  2. No α: if antipasto needs to support inference-time steering (the paper's
  selling point), thread an α scalar through the forward and S_eff = S + 
  α·delta_s; X = α·A/2.

  And one perf fix: kill _block_diag, use einsum.
